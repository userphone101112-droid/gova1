# UI i18n Reference

Complete reference for internationalization in GoVa: architecture, runtime, locale files, enforcement, CI, and workflows.

---

## Overview

GoVa uses a **registry-driven i18n system**. The UI registry (`src/platform/ui/registry/`) is the source of truth for user-facing copy. Each translatable UI identity maps to exactly one translation key; locale JSON files hold the strings for `en` and `ar`.

Supported locales:

| Locale | Code | Direction | Font (root layout) |
|--------|------|-----------|-------------------|
| English | `en` | `ltr` | Inter |
| Arabic | `ar` | `rtl` | Noto Sans Arabic |

Core modules live under `src/platform/ui/i18n/`.

---

## Architecture

```
UI Registry (identities + paths)
        │
        ▼
generateTranslationKeyFromUi()   ← binding rule
        │
        ▼
locales/<feature>/{en,ar}.json
        │
        ▼
getAppDictionary() / getDictionary()  ← merge + cache
        │
        ▼
I18nProvider (root layout)  →  useTranslation().t()
```

**Single root provider.** `I18nProvider` in `src/app/layout.tsx` loads the full app dictionary once at SSR. Route feature context updates via pathname (`resolveFeatureFromPathname`).

**Language SSOT.** User language is stored in the unified Zustand store (`src/store/unified.store.ts`) and persisted to:

- Cookie `gova-global-ssot` / `gova-unified-store` (JSON with `language`)
- Fallback cookie `locale`

Server reads locale via `getLocale()` in `src/platform/ui/i18n/utils/getLocale.server.ts`.

## UI Path → Translation Key

Registry paths use four segments when a role is present:

```ts
feature.section.role.element
```

Translation keys drop the **role** segment (`display`, `actions`, `form`, `layout`, etc.) and camelCase the element:

```ts
feature.section.camelCaseElement
```

| Registry path | Translation key |
|---------------|-----------------|
| `onboarding.welcome.display.title` | `onboarding.welcome.title` |
| `onboarding.welcome.actions.get-started` | `onboarding.welcome.getStarted` |
| `merchant.performance.display.revenue-today` | `merchant.performance.revenueToday` |
| `shared-layout.sidebar.actions.close-button` | `shared-layout.sidebar.closeButton` |

Implementation: `generateTranslationKeyFromUi()` in `src/platform/ui/i18n/binding/registry-binding.ts`.

Three-segment paths (`page.section.element`) are also supported; the element is taken from the third segment.

### Naming rules

- **Leaf keys** must be `camelCase` (validated by `i18n:validate`).
- **Section names** may stay kebab-case when they match feature structure.

Allowed:

```json
{ "currencyAud": "AUD ($)" }
```

Not allowed:

```json
{ "currency-aud": "AUD ($)" }
```

Section example:

```json
{
  "store-info": {
    "storeName": "Store Name"
  }
}
```

## Identities That Do Not Need Translations

Not every registry identity requires a locale entry. Skipped by `isTranslationRequiredForUiIdentity()` in `src/platform/ui/registry/registry.ts`:

| Condition | Example |
|-----------|---------|
| `category: 'container'` | Structural DOM wrappers (`onboarding.dom.*`) |
| Path contains `.dom.` | Auto-generated layout containers |
| Listed in `NO_TRANSLATION_REQUIRED` | Dynamic-only content (merchant name, avatar, banner) |
| `common.*` category paths | Shared non-text primitives (`isCategoryUiPath`) |

Current `NO_TRANSLATION_REQUIRED` entries:

- `merchant.hero.display.banner`
- `merchant.hero.display.avatar`
- `merchant.hero.display.name`

Add paths here when visible text comes from data, not locale files.

## Locale File Layout

Root: `src/platform/ui/i18n/locales/`

Each feature is a folder with paired locale files:

```
locales/
  auth/
    en.json
    ar.json
  common/
    en.json
    ar.json
  devtools/
  error-boundary/
  home/
  merchant/
  onboarding/
  settings/
  shared-layout/
  splash/
```

**Current features:** `auth`, `common`, `devtools`, `error-boundary`, `home`, `merchant`, `onboarding`, `settings`, `shared-layout`, `splash`.

JSON files nest under the feature namespace:

```json
{
  "onboarding": {
    "welcome": {
      "title": "Welcome to Gova! 🎉",
      "description": "Let's set up your merchant account...",
      "getStarted": "Get Started"
    }
  }
}
```

The dot-notation key `onboarding.welcome.title` resolves through this nested structure.

`common/en.json` holds shared namespaces: `navigation`, `buttons`, `validation`, `messages`, etc.

## Route → Feature Mapping

`I18N_ROUTE_MANIFEST` in `src/platform/ui/i18n/core/i18n-route-manifest.ts` maps URL prefixes to the active i18n feature (longest prefix wins):

| Route prefix | Feature |
|--------------|---------|
| `/devtools` | `devtools` |
| `/settings` | `settings` |
| `/onboarding` | `onboarding` |
| `/merchant` | `merchant` |
| `/notifications`, `/favorites`, `/orders`, `/profile` | `shared-layout` |
| `/home` | `home` |
| `/registration`, `/login` | `auth` |
| `/` | `splash` |
| (fallback) | `common` |

Used by `useTranslation()` for localization boundary checks and by the UI Inspector.

## Feature Scopes (Dictionary Merge)

`FEATURE_SCOPES` in `src/platform/ui/i18n/core/featureScope.ts` defines which namespaces merge for a route:

| Feature | Merged namespaces (order: later overrides) |
|---------|---------------------------------------------|
| `common` | `common`, `error-boundary` |
| `home` | `common`, `shared-layout`, `home` |
| `auth` | `common`, `auth` |
| `splash` | `common`, `splash` |
| `shared-layout` | `common`, `shared-layout` |
| `settings` | `common`, `settings` |
| `merchant` | `common`, `shared-layout`, `merchant` |
| `onboarding` | `common`, `shared-layout`, `onboarding` |

`getMergedDictionary(locale, feature)` deep-merges scoped files. `getAppDictionary()` merges all features listed in `APP_DICTIONARY_FEATURES` for the root layout SSR payload.

---

## Localization Boundaries

Features may only access their own namespace plus shared namespaces. Enforced at runtime by `validateTranslationKey()` / `enforceLocalizationBoundary()` in `src/platform/ui/i18n/core/enforceBoundary.ts`.

Allowed cross-feature namespaces:

`common`, `navigation`, `buttons`, `validation`, `messages`, `shared-layout`, `error-boundary`

Violations log a warning in development when `t()` is called with an out-of-scope key.

## Runtime API

### Client (`@/platform/ui`)

| Export | Purpose |
|--------|---------|
| `I18nProvider` | Root context provider |
| `useTranslation()` | Returns `{ t, locale, setLocale, feature }` |
| `useI18nContext()` | Raw context access |
| `LocaleProvider` | Syncs `lang` / `dir` on `<html>` when language changes |
| `t()` | Low-level dictionary lookup (prefer hook) |
| `boundTranslation()` | Resolve UI identity → key → string |
| `resolveTranslationKey()` | UI identity or key → `TranslationKey` |
| `generateTranslationKeyFromUi()` | Binding rule |
| `getDirection()` | `en` → `ltr`, `ar` → `rtl` |
| `Locale`, `Feature`, `TranslationKey` | Types |

### Server (`@/platform/ui/server`)

| Export | Purpose |
|--------|---------|
| `getLocale()` | Read language from SSOT cookie |
| `getAppDictionaryCached()` | Full merged dictionary for SSR |
| `getDictionary()` / `getDictionaryCached()` | Feature-scoped dictionary |
| `getDirection()` | Direction from locale |
| `discoverFeatures()` | List features from filesystem |

### `useTranslation().t(source, fallback?)`

Accepts either:

1. **UI identity object** from the registry (preferred): `t(ONBOARDING.WELCOME.TITLE)`
2. **Explicit translation key string**: `t('onboarding.welcome.title')`

Resolution flow (`resolveTranslationSource.ts`):

1. If UI identity → check translation required → `generateTranslationKeyFromUi(path)`
2. If string → use as key directly
3. Lookup in merged dictionary via `t()` in `core/t.ts`

Development behavior for missing keys:

- Console error (once per key)
- Visible marker: `[missing:key]`
- Hint to run `npm run i18n:sync` then `i18n:validate`

## Component Usage Pattern

```tsx
'use client';
import { ONBOARDING, useTranslation } from '@/platform/ui';

export function WelcomeStep() {
  const { t } = useTranslation();

  return (
    <h1 data-ui-uuid={ONBOARDING.WELCOME.TITLE.uuid}>
      {t(ONBOARDING.WELCOME.TITLE)}
    </h1>
  );
}
```

Rules:

- Pass the **registry identity object** to `t()`, not a hardcoded string key.
- Every user-visible element needs `data-ui-uuid` from the registry.
- Use logical CSS properties for RTL (`ms-`, `me-`, `text-start`, `text-end`) — enforced by ESLint.

### Language switching

`LanguageSwitcher` (`src/components/shared/LanguageSwitcher.tsx`) calls `setLocale('en' | 'ar')`, which:

1. Updates unified store
2. Reloads dictionary client-side
3. `LocaleProvider` updates `<html lang dir>`

## Type Safety

`npm run i18n:generate-keys` regenerates `src/platform/ui/i18n/translation-keys.d.ts`:

- Per-feature union types (`authTranslationKey`, `homeTranslationKey`, …)
- Combined `TranslationKey` / `AllTranslationKeys`

**Do not edit manually.** ESLint rule `validate-translation-keys` reads this file to validate string literal keys in `t("...")` calls.

## Commands

### Primary pipeline

```powershell
npm run i18n
```

Alias for `ci:i18n`:

```
registry:validate-uuids
  → i18n:sync
  → i18n:validate
  → i18n:validate-bindings
  → i18n:generate-keys
```

Also runs on `prebuild` and `ci:pre-commit`.

### Individual scripts

| Command | Script | What it does |
|---------|--------|--------------|
| `npm run i18n:sync` | `sync-translations.ts` | Adds missing keys from registry to `en.json` / `ar.json`. Never overwrites existing values. EN uses identity description; AR gets `TODO(ar): …` placeholder. |
| `npm run i18n:validate` | `validate-translations.ts` | Ensures every feature has both `en.json` and `ar.json` with identical key sets. Validates leaf camelCase naming. **Exits 1 on failure.** |
| `npm run i18n:validate-bindings` | `validate-registry-bindings.ts` | Every translatable registry identity has a matching locale key. **Exits 1 on failure.** |
| `npm run i18n:generate-keys` | `generate-translation-keys.ts` | Regenerates `translation-keys.d.ts` from `en.json` files. |
| `npm run i18n:prune-orphans` | `prune-orphans.ts` | Removes orphan keys reported by `find-orphans`. Pass `--apply` to write. |
| `npm run audit:orphans` | `find-orphans.ts` | Detects unused translations, orphan UI identifiers, missing language pairs, duplicates. |
| `npm run audit:unified-ui-i18n` | `audit-unified-ui-i18n.ts` | Full UI + i18n audit; writes reports under `docs/audits/`. **Exits 1 on failure.** Used in `ci:check`. |

### CI integration

| Pipeline step | i18n checks included |
|---------------|---------------------|
| `ci:pre-commit` | `ci:i18n` |
| `prebuild` | `ci:i18n` |
| `ci:check` | `ci:i18n` + `audit:unified-ui-i18n` + ESLint i18n rules via `lint` |
| `ci:enforcement` | `ci:i18n` + `audit:unified-ui-i18n` + `audit:orphans` |

## ESLint Enforcement

Plugin: `i18n-enforcement` (`src/platform/ui/enforcement/eslint/i18n-enforcement.js`), configured in `eslint.config.mjs`.

| Rule | Severity | Description |
|------|----------|-------------|
| `validate-translation-keys` | error | `t("key")` must exist in `translation-keys.d.ts` |
| `no-hardcoded-text` | error | No user-facing literals in JSX / attributes / object props |
| `no-decorative-with-text` | error | `DECORATIVE.*` identities must not carry visible text |
| `validate-ui-i18n-alignment` | error | UI identifier and translation key must share feature namespace |
| `require-ui-i18n-binding` | error | `Ui*` components with `ui=` prop need bound translation keys |
| `no-orphan-translations` | warn | Placeholder (full scan via `audit:orphans`) |
| `enforce-ui-translation-coupling` | warn | `t()` should be inside a UI context (`data-ui-uuid` or `Ui*`) |
| `no-directional-violations` | error | Block `ml-`, `mr-`, `text-left`, `left-`, etc.; use logical properties |

**Exceptions:**

- Test, spec, and story files are skipped.
- `src/platform/ui/devtools/**` — `no-hardcoded-text` is off (inspector uses `INSPECTOR_LABELS` constants and `devtools` locale namespace where needed).

**Migration mode:** set `MIGRATION_MODE=true` to downgrade i18n ESLint errors to warnings.

### Standalone ESLint UI tests

```powershell
npm run test:eslint-ui
```

Runs UUID enforcement tests (not i18n-specific, but part of `ci:check` alongside lint).

## Adding a New Feature

### Option A — scaffold script

```powershell
npx tsx scripts/generate-feature.ts my-feature
```

Creates:

- `src/platform/ui/registry/features/my-feature.ts`
- `src/platform/ui/i18n/locales/my-feature/{en,ar}.json`
- Feature page/layout scaffolds under `src/features/my-feature/`

Then:

1. `npm run registry:materialize-uuids && npm run registry:generate`
2. Add route entry to `I18N_ROUTE_MANIFEST`
3. Add scope to `FEATURE_SCOPES`
4. Add feature to `APP_DICTIONARY_FEATURES` in `getDictionary.ts`
5. `npm run i18n`

### Option B — manual workflow

1. Register UI identities in `src/platform/ui/registry/features/<feature>.ts`
2. Create `src/platform/ui/i18n/locales/<feature>/en.json` and `ar.json`
3. Use identities in components with `t(IDENTITY_OBJECT)`
4. Run `npm run i18n:sync` to fill missing keys
5. Translate Arabic placeholders (`TODO(ar): …`)
6. Run `npm run i18n` before commit/build

## Typical Workflows

### New UI element with copy

1. Add identity to registry feature file (or run registry generator).
2. `npm run i18n:sync` — adds EN + AR stub keys.
3. Edit `en.json` / `ar.json` with final copy.
4. In component: `data-ui-uuid={FEATURE.SECTION.ELEMENT.uuid}` + `{t(FEATURE.SECTION.ELEMENT)}`.
5. `npm run i18n` — validate, bind, regenerate types.

### Fix missing translation at runtime

Development console shows:

```
[i18n] Missing translation key: "feature.section.key". Run npm run i18n:sync, then npm run i18n:validate.
```

Run sync, fill values, validate.

### Remove unused keys

```powershell
npm run audit:orphans          # inspect report
npm run i18n:prune-orphans     # dry run
npm run i18n:prune-orphans -- --apply
npm run i18n:generate-keys
```

Protected keys from registry bindings are not pruned (`getRequiredRegistryBindingKeys()`).

## Devtools & Inspector Notes

- UI Inspector resolves page feature via `resolveFeatureFromPathname`.
- Internal inspector labels use hardcoded `INSPECTOR_LABELS` (`inspector-labels.ts`) to avoid key-collision edge cases — not app-facing copy.
- Inspector components under `src/platform/ui/devtools/` are exempt from `no-hardcoded-text`.
- `devtools` locale namespace exists for inspector UI that does use `t()`.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---------|--------------|-----|
| `[missing:key]` in UI | Key not in JSON or typo | `npm run i18n:sync`, check key shape |
| `i18n:validate` fails on AR | Missing/extra keys vs EN | Align both locale files per feature |
| `validate-bindings` fails | Registry identity without locale entry | `i18n:sync` or add key manually |
| ESLint `invalidTranslationKey` | Stale types | `npm run i18n:generate-keys` |
| Boundary warning in console | Cross-feature `t('otherFeature.key')` | Use own feature or `common.*` / `shared-layout.*` |
| Language not persisting | SSOT cookie missing | Check unified store persist + `locale` cookie |
| RTL layout broken | Physical CSS directions | Use `ms-`/`me-`/`start`/`end`; run lint |
