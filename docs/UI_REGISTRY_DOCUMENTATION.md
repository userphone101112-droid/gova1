## Overview

The UI Registry System is a comprehensive architecture for managing UI element identities across the application. It provides a centralized, type-safe registry that couples UI components with their metadata, translations, and source locations. This system ensures consistency, enables powerful debugging capabilities, and facilitates analytics tracking.

### Key Benefits
- **Centralized Management**: All UI element identities defined in one location
- **Type Safety**: TypeScript-based type enforcement prevents mismatches
- **Translation Coupling**: Automatic binding between UI identifiers and i18n keys
- **Source Tracking**: Automatic mapping to source file locations
- **Telemetry Ready**: Built-in support for analytics and error tracking
- **Validation**: Runtime and compile-time validation ensures integrity

## Core Components

### 1. UI Registry (`src/shared/ui-registry.ts`)

The central registry that defines all UI element identities in the application.

**Key Features:**
- Defines UI identities with structured metadata
- Provides lookup maps for quick access
- Validates registry integrity at load time
- Supports deprecation warnings
- Enables backward compatibility with legacy string-based identifiers

### 2. Registry Binding (`src/shared/unified-ui-i18n/registry-binding.ts`)

Enforces structural coupling between UI identifiers and translation keys.

**Key Features:**
- Validates UI identifier format
- Generates expected translation keys
- Detects cross-feature mapping violations
- Identifies orphan UI identifiers and translations
- Provides binding validation results

### 3. Bound UI Hooks (`src/shared/unified-ui-i18n/useBoundUI.ts`)

Type-safe React hooks for using bound UI components.

**Key Features:**
- Enforces UI-translation coupling at type level
- Validates translation key existence
- Provides context-aware translation functions
- Validates component-level bindings

### 4. UI Source Index (`src/shared/ui-source-index.ts`)

Auto-generated mapping of UI IDs to their source locations.

**Key Features:**
- Maps each UI identity to its source file
- Tracks component name and line number
- Enables reverse lookup for debugging
- Generated by audit scripts

### 5. UI Telemetry (`src/shared/ui-telemetry.ts`)

Telemetry extraction layer for analytics and inspector tools.

**Key Features:**
- Tracks UI interactions, views, and errors
- Resolves element identities from DOM elements
- Provides source location information
- Development-mode logging

## Data Structures

### UiIdentity Interface

```typescript
export interface UiIdentity {
  readonly id: string;                    // Stable unique identifier (e.g., "UI_HOME_PROMO_BANNER_ACTION")
  readonly path: string;                  // Hierarchical path (e.g., "home.promo-banner.actions.banner-button")
  readonly description: string;           // Human-readable description
  readonly category: 'action' | 'input' | 'navigation' | 'display' | 'container';
  readonly feature: string;               // Feature/module name (e.g., "home")
  readonly version: string;               // Semantic version
  readonly createdAt: string;             // ISO timestamp
  readonly updatedAt: string;             // ISO timestamp
  readonly deprecated?: boolean;          // Optional deprecation flag
}
```

### UiSourceLocation Interface

```typescript
export interface UiSourceLocation {
  id: string;              // UI identity ID
  path: string;            // UI identity path
  sourceFile: string;      // Source file path
  sourceComponent: string; // Component name
  sourceLine: number;      // Line number in source
  feature: string;         // Feature name
}
```

### BindingRecord Interface

```typescript
export interface BindingRecord {
  ui: UiIdentifier;              // UI identifier path
  translationKey: string;       // Corresponding translation key
  feature: string;              // Feature name
  page: string;                 // Page name
  section: string;              // Section name
  component: string;            // Component name
  element: string;              // Element name
}
```

## How It Works

### 1. Registration Phase

UI elements are registered in `ui-registry.ts` using a hierarchical structure:

```typescript
export const HOME = {
  PROMO_BANNER: {
    ACTION_BUTTON: {
      id: 'UI_HOME_PROMO_BANNER_ACTION',
      path: 'home.promo-banner.actions.banner-button',
      description: 'Promo banner call-to-action button',
      category: 'action',
      feature: 'home',
      version: '1.0.0',
      createdAt: '2026-06-15',
      updatedAt: '2026-06-15',
    } as const,
  },
} as const;
```

### 2. Index Generation

The `ui-source-index.ts` file is auto-generated by audit scripts that scan the codebase for UI registry usage and map each ID to its source location.

### 3. Component Usage

Components use UI-identified components that accept a `ui` prop:

```typescript
import { HOME } from '@/shared/ui-registry';
import { UiButton } from '@/components/ui';

<UiButton ui={HOME.PROMO_BANNER.ACTION_BUTTON}>
  {t('home.promoBanner.cta')}
</UiButton>
```

### 4. Runtime Resolution

The `UiButton` component resolves the UI identity and adds data attributes:

```typescript
const identity = resolveUiParam(ui);
validateRuntimeIdentity('UiButton', ui, identity);

<Button
  data-ui-id={identity?.id}
  data-ui-path={identity?.path}
  data-ui-feature={identity?.feature}
  {...props}
/>
```

### 5. Validation

The registry is automatically validated at module load time:
- Checks for duplicate identifiers
- Validates naming convention (page.section.component.element)
- Ensures no duplicate stable IDs

## UI-i18n Binding System

### Naming Convention

**UI Identifier Format:** `page.section.component.element`
- Example: `home.promo-banner.actions.banner-button`

**Translation Key Format:** `page.section.element` (kebab-case to camelCase)
- Example: `home.promoBanner.bannerButton`

### Binding Rules

1. **Format Validation**: UI identifiers must follow the 4-part dot notation
2. **Feature Alignment**: Translation keys must belong to the same feature as the UI identifier
3. **Naming Convention**: Translation keys should follow the expected pattern (with warnings for mismatches)
4. **No Orphans**: All UI identifiers must have corresponding translations

### Binding Functions

#### generateTranslationKeyFromUi

```typescript
export function generateTranslationKeyFromUi(ui: UiIdentifier): string {
  const parts = ui.split('.');
  const page = parts[0];
  const section = parts[1];
  const element = parts[3]; // Skip component part
  
  // Convert kebab-case to camelCase
  const camelCaseElement = element.replace(/-([a-z])/g, (_, c) => c.toUpperCase());
  
  return `${page}.${section}.${camelCaseElement}`;
}
```

## Telemetry System

### Tracking Functions

#### trackUiInteraction

Tracks user interactions with UI elements:

```typescript
export function trackUiInteraction(target: HTMLElement | UiParam | null): UiTelemetryData | null {
  const identity = resolveTelemetryIdentity(target);
  if (!identity) return null;

  return {
    id: identity.id,
    path: identity.path,
    feature: identity.feature,
    timestamp: new Date().toISOString(),
  };
}
```

#### trackUiView

Tracks element visibility/exposure:

```typescript
export function trackUiView(target: HTMLElement | UiParam | null): UiTelemetryData | null {
  // Similar structure to trackUiInteraction
}
```

#### trackUiError

Tracks errors occurring on UI elements:

```typescript
export function trackUiError(target: HTMLElement | UiParam | null, error: Error): UiErrorTelemetryData | null {
  return {
    id: identity.id,
    path: identity.path,
    feature: identity.feature,
    timestamp: new Date().toISOString(),
    errorMessage: error.message,
    errorStack: error.stack,
  };
}
```

### Inspector Integration

The `resolveElementIdentity` function provides source location information for debugging:

```typescript
export function resolveElementIdentity(target: HTMLElement | UiParam | null): InspectorLocationInfo | null {
  const identity = resolveTelemetryIdentity(target);
  const sourceLocation = UI_SOURCE_INDEX[identity.id];

  return {
    ...identity,
    sourceFile: sourceLocation?.sourceFile || 'unknown',
    sourceComponent: sourceLocation?.sourceComponent || 'unknown',
    sourceLine: sourceLocation?.sourceLine || 1,
  };
}
```

## Best Practices

### 1. Registration

- **Use hierarchical organization**: Group related UI elements together
- **Follow naming convention**: Use kebab-case for all path segments
- **Provide clear descriptions**: Help other developers understand the purpose
- **Set appropriate categories**: Choose the most accurate category for the element
- **Version your changes**: Update version numbers when modifying identities

### 2. Usage

- **Always use registered identities**: Never create ad-hoc UI identity objects
- **Import from registry**: Use the exported constants (e.g., `HOME.PROMO_BANNER.ACTION_BUTTON`)
- **Use UI-identified components**: Prefer `UiButton` over `Button` for tracked elements
- **Validate in development**: Leverage runtime validation during development

### 3. Translation Binding

- **Follow naming convention**: Ensure translation keys match the expected pattern
- **Keep features aligned**: Don't map UI identifiers to translations from other features
- **Handle common translations**: Use the 'common' feature for shared translations
- **Validate bindings**: Use the binding validation functions before deployment

### 4. Deprecation

- **Mark deprecated items**: Set the `deprecated` flag when removing UI elements
- **Update references**: Replace deprecated identities throughout the codebase
- **Monitor usage**: Check for orphan identifiers before removing them

### 5. Source Index

- **Regenerate after changes**: Run the audit script after modifying the registry
- **Commit the index**: Include the auto-generated index in version control
- **Verify mappings**: Check that source locations are accurate after refactoring

## Validation & Error Handling

### Registry Validation

The registry is automatically validated at module load time:

```typescript
validateRegistry();
```

This checks for:
- Duplicate UI paths
- Duplicate stable IDs
- Invalid identifier formats

### Runtime Validation

Components validate UI identities at runtime:

```typescript
validateRuntimeIdentity('UiButton', ui, identity);
```

This checks for:
- Unknown UI identities
- Broken UI identity mappings
- Deprecated identities

### Binding Validation

Validate UI-translation bindings:

```typescript
const validation = validateBinding(
  ui,
  translationKey,
  availableTranslationKeys
);

if (!validation.isValid) {
  console.error('Binding errors:', validation.errors);
}
```

### Error Types

#### Missing Translation
UI identifier exists but translation key is missing.

#### Missing UI Identifier
Translation key exists but UI identifier is not registered.

#### Cross-Feature Mapping
UI identifier and translation key belong to different features.

#### Invalid Format
UI identifier does not follow the required format.

#### Orphan Translation
Translation key exists but is not used by any UI element.

#### Orphan UI Identifier
UI identifier is registered but never used in any component.

## Development Workflow

### Adding a New UI Element

1. **Register the UI Identity**
   ```typescript
   export const HOME = {
     NEW_SECTION: {
       NEW_BUTTON: {
         id: 'UI_HOME_NEW_SECTION_BUTTON',
         path: 'home.new-section.actions.new-button',
         description: 'New section button',
         category: 'action',
         feature: 'home',
         version: '1.0.0',
         createdAt: '2026-06-15',
         updatedAt: '2026-06-15',
       } as const,
     },
   } as const;
   ```

2. **Add Translation**
   ```json
   {
     "home": {
       "newSection": {
         "newButton": "Click Here"
       }
     }
   }
   ```

3. **Use in Component**
   ```typescript
   import { HOME } from '@/shared/ui-registry';
   import { UiButton } from '@/components/ui';

   <UiButton ui={HOME.NEW_SECTION.NEW_BUTTON}>
     {t('home.newSection.newButton')}
   </UiButton>
   ```

4. **Regenerate Source Index**
   ```bash
   npm run audit:ui-i18n
   ```

### Modifying an Existing UI Element

1. **Update the registry entry**
2. **Update version number**
3. **Update translation if needed**
4. **Search and replace usages**
5. **Regenerate source index**
6. **Run validation**

### Deprecating a UI Element

1. **Set deprecated flag**
   ```typescript
   {
     id: 'UI_OLD_BUTTON',
     // ...
     deprecated: true,
   }
   ```

2. **Replace usages throughout the codebase**

3. **Remove after deprecation period**

### Debugging UI Issues

1. **Check browser console for validation errors**
2. **Use inspector to view data attributes**
3. **Resolve element identity to find source location**
4. **Check telemetry logs for interaction tracking**
5. **Validate bindings with validation functions**

## Summary

The UI Registry System provides a robust, type-safe architecture for managing UI element identities across the application. By centralizing UI definitions, enforcing translation bindings, and providing telemetry capabilities, it ensures consistency, enables powerful debugging, and facilitates analytics tracking.

The system is designed to be:
- **Type-safe**: TypeScript enforcement prevents mismatches
- **Validated**: Runtime and compile-time validation ensures integrity
- **Traceable**: Source location mapping enables easy debugging
- **Extensible**: Easy to add new UI elements and features
- **Maintainable**: Clear structure and validation support long-term maintenance

Following the best practices outlined in this document will ensure a smooth development experience and a reliable, maintainable UI registry system.
